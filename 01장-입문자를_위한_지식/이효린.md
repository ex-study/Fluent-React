> `<button id="likeButton" data-liked="false">좋아요</button>`
> 
- data-liked는 data-slot이라고 부르는 것인가? attribute인가?

정식 명칭은 **data attribute (데이터 속성)** 이다.

data-slot이라는 별도의 개념이 있는 것이 아니라,

**data- 로 시작하는 모든 attribute를 data attribute라고 부른다.**

즉,

- data-liked
- data-id
- data-user-name

모두 data attribute이다.

이는 **HTML 요소에 추가적인 데이터를 저장하기 위해 사용하는 속성이다.**

> <button data-liked="false">
> 
- data-liked는 왜 사용하는 것인가?

**버튼의 상태를 저장하기 위해 사용하는 것이다.**

예:

```
data-liked="false"
```

→ 좋아요가 눌리지 않은 상태라는 의미이다.

```
data-liked="true"
```

→ 좋아요가 눌린 상태라는 의미이다.

즉, **UI 상태를 DOM에 저장하기 위한 용도이다.**

> likeButton.setAttribute("data-liked", !liked);
> 
- setAttribute는 setter 함수인가? 무엇인가?

**DOM 요소의 attribute 값을 변경하는 setter 함수이다.**

형식은 다음과 같다.

```
element.setAttribute("속성명", "값");
```

예:

```
likeButton.setAttribute("data-liked", "true");
```

→ data-liked 값을 true로 변경했다는 의미이다.

즉, **HTML attribute 값을 변경하는 함수이다.**

> setAttribute
> 
- setAttribute는 비동기로 동작하는 것인가?

**비동기가 아니라 동기 함수이다.** setAttribute를 실행하는 즉시 attribute 값이 변경된다.

예:

```
button.setAttribute("data-liked", "true");
console.log(button.getAttribute("data-liked"));
```

결과:

```
true
```

즉시 변경되었다.

Promise, async 같은 비동기 처리가 아니다.

> setAttribute 와 useState setter
> 
- setter 함수는 왜 비동기로 동작하는 것처럼 보이는 것인가?

**React의 setter는 실제 비동기가 아니라, 업데이트를 예약하는 방식이기 때문이다.**

React는 성능 최적화를 위해 state 변경을 즉시 수행하지 않고,

여러 변경을 모아서 한 번에 렌더링한다.

이를 batching이라고 한다.

예:

```
setCount(1)
setCount(2)
setCount(3)
```

React는 3번 렌더링하지 않고 1번만 렌더링한다. 따라서 비동기처럼 보이는 것이다.

- React setter 호출 후 값이 바로 변경되지 않는 이유는 무엇인가?

**state 변경이 즉시 적용되지 않고, 다음 렌더링 시 적용되기 때문이다.**

예:

```
setCount(1)
console.log(count)
```

출력:

```
0
```

아직 렌더링이 발생하지 않았기 때문이다.

> 
> 
> 
> ```jsx
> var xhr = new XMLHttpRequest();
> 
> xhr.open("POST", "/like", true);
> xhr.setRequestHeader("Content-Type", "application/json;charset=UTF-8");
> xhr.onload = function () {
>   if (xhr.status >= 200 && xhr.status < 400) {
>     // 성공!
>     likeButton.setAttribute("data-liked", !liked);
>     likeButton.textContent = liked ? "Like" : "Liked";
>   } else {
>     // 서버까지 데이터는 보냈지만 실패가 반환됨
>     console.error("Server returned an error:", xhr.statusText);
>   }
> };
> xhr.onerror = function () {
>   // 접속 관련 오류가 발생
>   console.error("네트워크 오류");
> };
> xhr.send(JSON.stringify({ liked: !liked }));
> ```
> 
- **xhr 객체는 어떻게 구성되어 있는 것인가?**

xhr 객체는 **브라우저가 제공하는 XMLHttpRequest 클래스의 인스턴스이다.**

즉,

```
var xhr = new XMLHttpRequest();
```

위 코드는  HTTP 요청을 보내기 위한 객체를 생성한 것이다.

이 객체 안에는

- 요청 설정
- 요청 전송
- 응답 처리

를 위한 **property + method + event handler** 로 구성되어 있다.

구조적으로는 다음 3가지 영역으로 나뉜다.

1. 요청 설정
2. 요청 실행
3. 응답 처리

- **xhr 객체 안에는 onerror 말고 어떤 메서드와 속성이 있는 것인가?**

대표적인 것들은 다음과 같다.

**요청 설정 관련**

**open**

```
xhr.open(method, url, async)
```

요청 초기화 메서드이다.

**setRequestHeader**

```
xhr.setRequestHeader(name, value)
```

HTTP header 설정 메서드이다.

**요청 실행**

**send**

```
xhr.send(body)
```

요청을 서버로 전송하는 메서드이다. 가장 중요한 메서드이다.

**abort**

```
xhr.abort()
```

요청 취소 메서드이다.

**응답 데이터 관련 property**

**status**

```
xhr.status
```

HTTP 상태 코드이다.

예: 200 404 500

**responseText**

```
xhr.responseText
```

서버 응답 데이터이다.

문자열 형태이다.

**response**

```
xhr.response
```

응답 데이터이다. responseType에 따라 다르다.

정리하면 xhr는 요청 설정 요청 실행 응답 처리 기능을 모두 가진 객체이다.

- axios는 네트워크 통신 코드를 간단하게 실행시킬 수 있도록 해주는 라이브러리인가?

맞다. axios는 **XMLHttpRequest를 기반으로 만들어진 HTTP client 라이브러리이다.**

즉, XMLHttpRequest를 직접 사용하지 않고 더 간단하게 HTTP 요청을 작성할 수 있게 해준다.

예: XMLHttpRequest

```jsx
var xhr = new XMLHttpRequest();

xhr.open("POST", "/like");

xhr.onload = function () {
 console.log(xhr.responseText);
};

xhr.send();
```

axios

```jsx
axios.post('/like')
 .then(res => console.log(res.data))
```

훨씬 간단하다.

- axios는 내부적으로 무엇을 사용하는가?

브라우저에서는 XMLHttpRequest를 사용한다. Node.js에서는 http module을 사용한다.

즉, axios는 XMLHttpRequest를 감싼 wrapper이다.

p.37

> 우리가 작성하는 애플리케이션은 부작용(side effect)으로 가득 차게 되어 사용자 환경에 성패가 갈립니다.
> 
- 여기서 말하는 부작용(side effect)은 useEffect를 의미하는 것인가?
    
    useEffect 자체를 의미하는 것이 아니라, 일반적인 프로그래밍 개념의 side effect를 의미하는 것이다. side effect란 함수가 자신의 외부 상태를 변경하는 것을 의미하며, 대표적인 예로 DOM 직접 변경, 서버 요청, 전역 변수 변경 등이 있다. React의 useEffect는 side effect를 발생시키는 함수가 아니라, 이러한 side effect를 React의 렌더링 흐름 안에서 안전하게 실행하고 관리하기 위한 메커니즘이다. 즉, side effect는 문제 상황이고, useEffect는 그것을 통제하기 위한 해결 수단이다.
    

> 어떤 식으로든 일괄 처리해 한꺼번에 DOM에 적용하는 것입니다.
> 
- DOM을 일괄 처리한다는 것은 무엇인가?
    
    여러 번 DOM을 직접 수정하는 대신, 변경 사항을 메모리에서 먼저 처리한 후 한 번에 DOM에 반영하는 방식이다. 예를 들어 반복문 안에서 append를 여러 번 호출하는 대신, DocumentFragment 같은 임시 컨테이너에 element를 모아두고 마지막에 한 번만 DOM에 추가하는 것이다. 이렇게 하면 reflow 발생 횟수를 줄일 수 있어 성능이 개선된다. React의 Virtual DOM도 이러한 일괄 처리 개념을 기반으로 동작한다.

> 이 예시는 사용자 인터페이스에 데이터를 바인딩하고 이 데이터 바인딩을 사용해 페이지를 벗어나지 않은 채 사용자 인터페이스를 업데이트합니다.
> 
- 데이터 바인딩이란 무엇인가?
    
    데이터 바인딩이란 데이터와 UI를 연결하여, 데이터 상태에 따라 UI가 변경되도록 만드는 것을 의미한다. 예를 들어 좋아요 상태 값에 따라 버튼의 text를 변경하는 것이 데이터 바인딩이다. 즉, UI를 직접 수정하는 것이 아니라, 데이터를 기준으로 UI를 동기화하는 개념이다. 이는 상태 기반 UI(state-driven UI)의 핵심 개념이다.
    
- 페이지를 벗어나지 않은 채 사용자 인터페이스를 업데이트하는 것은 무엇인가 ?
    
    브라우저가 페이지를 새로고침하거나 이동하지 않고 현재 페이지에서 화면 일부만 변경된다는 의미이다.
    
    즉, 전체 페이지를 다시 불러오는 대신 데이터만 변경하고, 그 데이터에 연결된 UI가 자동으로 갱신된다. 예를 들어 좋아요 버튼을 눌렀을 때 페이지가 새로고침되지 않고 버튼 텍스트만 변경되는 것이 이에 해당한다.
    

p.39

> jQuery는 사용자 인터페이스 자체를 직접 조작합니다.
> 
- 직접 조작한다는 것은 무엇인가?
    
    state를 변경하는 방식이 아니라, DOM element를 직접 선택해서 수정하는 것을 의미한다. 예를 들어 `$(”#likeButton”).text(“좋아요!!”)`는 state를 변경하는 것이 아니라 DOM을 직접 변경한 것이다. 이 방식은 간단하지만, 애플리케이션 규모가 커질수록 상태 추적이 어려워지고 예측하기 어려운 문제가 발생한다. React는 이와 달리 state를 변경하면 React가 DOM을 업데이트하는 구조를 사용한다.
    

> 페이지 일부를 변경하면 예상하지 못한 방식으로 다른 부분에도 영향을 미쳐 상호 작용이 복잡해지고 동작을 파악하기가 난해해집니다.
> 
- 왜 예상하지 못한 영향이 발생하는 것인가?
    
    DOM은 전역에서 공유되는 객체이기 때문이다. 즉, 여러 JavaScript 코드가 동일한 DOM을 수정할 수 있기 때문에 한 코드의 변경이 다른 코드의 동작에 영향을 줄 수 있다. 이러한 구조에서는 현재 UI 상태가 어떻게 만들어졌는지 추적하기 어려워지고, 버그 발생 시 원인 파악이 어려워진다.
    

> 최신 프레임워크는 직접적인 DOM 조작 없이 UI를 업데이트하는 구조적이고 예측 가능한 방법을 제공합니다.
> 
- 직접 DOM 조작 없이 어떻게 UI를 업데이트하는 것인가?
    
    state를 변경하면 프레임워크가 DOM 업데이트를 대신 수행하는 방식이다. 예를 들어 React에서는 element를 직접 수정하지 않고 setState 또는 useState setter를 호출하여 state를 변경한다. 그러면 React가 이전 state와 새로운 state를 비교한 후 필요한 DOM만 업데이트한다. 이 방식은 UI가 항상 state의 결과로 결정되기 때문에 예측 가능성이 높고 유지보수가 쉬워진다.
    

> 사용자 인터페이스의 소유권을 브라우저와 공유합니다.
> 
- UI 소유권을 브라우저와 공유한다는 것은 무엇인가?
    
    UI 상태를 JavaScript와 브라우저가 함께 관리하는 구조라는 의미이다. jQuery 방식에서는 DOM이 실제 상태 저장소 역할을 하며, JavaScript는 그 DOM을 직접 수정한다. 즉, 상태의 진짜 source of truth가 DOM에 있다. 반면 React에서는 state가 source of truth이고, DOM은 그 결과물일 뿐이다. 이 차이가 구조적 안정성과 예측 가능성의 차이를 만든다.
    

> 단방향 데이터 흐름 패러다임을 따르지 않는 것은 당시 라이브러리에서는 흔한 문제였습니다.
> 
- 단방향 데이터 흐름이란 무엇인가?
    
    데이터가 한 방향으로만 흐르는 구조를 의미한다. React에서는 state가 변경되면 UI가 변경되지만, UI가 직접 state를 변경하지는 않는다. 즉, state → UI 방향으로만 흐른다. 반면 jQuery는 UI를 직접 수정하고, 그 UI를 기반으로 다시 로직이 동작하는 구조이므로 흐름이 양방향으로 발생할 수 있다. 이로 인해 상태 관리가 복잡해지고 버그 발생 가능성이 높아진다.

p.42

> MVC 아키텍처를 사용하면 복잡한 사용자 인터페이스를 관리할 때 어려움을 겪곤 합니다. 상태 변경과 UI의 다양한 부분에 미치는 영향을 관리하기가 어려워질 수 있습니다.
> 
- 예시가 있었으면 좋겠다.

**예시: 사용자 이름 변경 기능 (MVC)**

**화면 구성**

```jsx
<div id="headerName"></div>
<div id="profileName"></div>
<input id="nameInput" />
<button id="changeBtn">이름 변경</button>
```

사용자 이름은 헤더에도, 프로필에도 표시된다.

**Controller 코드 (MVC 방식)**

```
// Controller
$("#changeBtn").click(function () {
  var newName = $("#nameInput").val();

  UserModel.name = newName;

  // UI 각각 직접 수정
  $("#headerName").text(newName);
  $("#profileName").text(newName);
});
```

**처음에는 문제 없음**

정상 동작:

```
header: 김철수
profile: 김철수
```

**그런데 UI가 하나 더 추가됨**

```
<div id="sidebarName"></div>
```

**문제 발생**

Controller를 수정 안 하면

```
header: 김철수
profile: 김철수
sidebar: 이전 이름 (홍길동)
```

→ UI 상태 불일치 발생, Controller가 **UI 여러 곳을 각각 직접 관리하기 때문이다.**

즉,

```
state 변경
↓
header 수정 (직접)
profile 수정 (직접)
sidebar 수정 (직접)
```

하나라도 빠지면 버그 발생한다.

**더 큰 문제: Controller 충돌**

Controller가 여러 개 있는 경우

```
// Controller A
UserModel.name = "김철수";
$("#headerName").text("김철수");

// Controller B
UserModel.name = "이영희";
$("#profileName").text("이영희");
```

결과:

```
header: 김철수
profile: 이영희
```

→ 상태 불일치, 관리 어려움

**React 방식에서는** `setName("김철수")` 를 실행하면 React가 `header` `profile`
`sidebar` 를 자동으로 모두 업데이트한다.

**jQuery**

**개념**

- 2006년에 등장한 **DOM 조작 라이브러리**
- 핵심 목적: 브라우저 호환성 문제 해결 + DOM 제어 간소화
- document.querySelector 같은 복잡한 코드를 $()로 단순화

**예시**

```
$('#btn').click(function() {
  $('#text').hide();
});
```

**특징**

- DOM 선택, 이벤트 처리, AJAX 쉽게 처리
- 프레임워크가 아니라 **유틸 라이브러리**

**단점**

- 상태 관리 개념 없음
- DOM을 직접 조작 → 규모 커지면 관리 지옥
- UI 상태와 데이터 동기화 어려움
- 구조 없음 → 코드 스파게티화

**Backbone.js**

**개념**

- 2010년 등장

## 1.3 리액트 등장

p.57

> 대규모 애플리케이션상 잠재적 성능 문제 등 몇 가지 단점이 있었습니다.
> 

양방향 데이터 바인딩(Angular 방식)은 규모가 커질수록 성능과 상태 추적이 어려웠다.

- **양방향**은 데이터 ↔ UI가 서로 계속 영향을 주는 구조.
    
    예: 입력창 값이 바뀌면 모델이 바뀌고, 모델이 바뀌면 다른 UI도 다시 바뀜.
    
- 규모가 커지면 “어디서 값이 바뀌었는지” 흐름이 많아지고, 프레임워크가 변경 감지(체크)를 넓게 돌려야 해서 **느려지거나 추적이 어려워질 수 있음**.

> 리액트는 단방향 데이터 흐름 패턴을 도입했습니다.
> 

데이터 흐름을 한 방향으로 제한해 상태 예측과 디버깅을 쉽게 만들었다.

- state(데이터)가 위에서 아래로 내려가며 UI를 만든다: state → UI
- UI에서 변경이 필요하면 이벤트(클릭/입력)가 발생하고, 그 이벤트가 setState 같은 업데이트를 호출해서 state를 바꿈.
- state → UI → event → state
    
    UI가 state를 “직접” 바꾸는 게 아니라, 이벤트를 통해 바꾸는 느낌
    

> 앵귤러JS는 종종 DOM을 직접 조작했는데, 이는 성능 문제를 일으키기도 했습니다.
> 

직접 DOM 조작은 성능 저하와 상태 불일치 문제를 유발했다

- jQuery처럼 document.querySelector(...).textContent = ...로 여기저기 DOM을 직접 바꾸면,
    - 같은 정보를 표시하는 곳이 여러 군데일 때 **한 군데만 업데이트하고 나머지는 놓치는** 일이 생김 → **상태 불일치**
- 또 DOM을 자주 건드리면 브라우저가 레이아웃 계산/그리기 같은 작업을 반복해서 하게 되어 **비용이 커질 수 있음**(특히 변경이 많을 때).

> 리액트는 DOM에 대한 선언적 추상화를 제공합니다.
> 

“어떻게”가 아니라 “무엇을 보여줄지”만 선언한다.

- 명령형(어떻게): “버튼 텍스트를 바꿔라, class를 붙여라…” 처럼 **조작 절차를 나열**
- 선언형(무엇): “liked가 true면 ‘Liked’, 아니면 ‘Like’가 보여야 한다”처럼 **결과 규칙만 적음**
- 리액트에서는 보통 이렇게 **‘결과’만 적고**, DOM 변경은 리액트가 알아서 처리:
    
    `return liked ? <LikedUI /> : <LikeUI />`
    

> 우리가 무엇을 완료하고 싶은지 기술하면 그 목표를 어떻게 이룰지는 리액트가 다룹니다.
> 

UI 결과만 정의하면 실제 DOM 업데이트는 리액트가 처리한다.

- 개발자는 “UI는 state의 함수(function)다”처럼 작성:
    
    UI = f(state)
    
- 그러면 state가 바뀔 때마다 리액트가 “그럼 화면은 이렇게 되어야겠네” 하고 다시 계산하고, **실제 화면(DOM) 반영은 내부에서 처리**해줌.

**불변 상태 (Immutability)**

> 리액트의 설계 철학은 애플리케이션 상태를 불변 값의 집합으로 기술하는 패러다임을 강조합니다.
> 

상태를 직접 변경하지 않고 새 상태를 만든다.

- 예: `state.user.name = "A"` 처럼 **기존 객체를 수정하지 않고**,
    
    `{ ...state, user: { ...state.user, name: "A" } }`처럼 **새 객체를 만들어 교체**
    
- 이게 중요한 이유: “변경이 일어났는지”를 **참조(주소) 비교로 빠르게 판단**하기 쉬움.

> 상태가 변경되면 원래 있던 상태를 직접 변경하는 대신 새로운 상태를 표현하는 새 객체를 반환합니다.
> 

이전 상태는 유지하고 새로운 상태 객체를 생성한다.

- 리액트는 보통 “이전 상태와 새 상태가 같은 객체인가?”(참조 동일성)로 **변경 여부를 빠르게 캐치**하는 최적화가 많다.
- 그래서 “그 자리에서 수정(mutating)”하면 변경을 놓치거나 추적이 어려워질 수 있어서, 아예 새 객체를 만드는 패턴이 잘 맞다.

> 불변성 덕분에 시간 여행 디버깅이 가능합니다.
> 

과거 상태로 되돌아가는 디버깅이 가능하다.

- 상태를 매번 새로 만들면, 과거 상태들이 그대로 남아(원본이 안 망가져)
    
    **이전 상태로 되돌아가서 UI를 재현**하기 쉽다.
    
- 예: Redux DevTools처럼 “n번째 액션 시점 상태로 이동” 같은 게 가능해지는 원리.

## 1장을 마무리하며

1장은 “왜 리액트가 필요했는가?”를 이해하기 위해 FE 개발 방식의 변화를 보여준 장이라고 생각합니다. 프론트엔드 개발을 리액트로 시작했던 사람으로써, jQuery, Angular, BackBone, Knockout에 대해서 대략적으로 이해할 수 있던 장이었고, 이러한 역사를 이해함으로써 앞으로의 발전 방향성을 예측할 수 있지 않을까? 라는 생각도 들었습니다. 

jQuery의 DOM 직접 조작 문제를 개선하기 위해 나타난 Backbone(MVC), Knockout(MVVM), AngluarJS(양방향 바인딩) 프레임워크들 또한 각각의 한계점을 드러냈습니다. 

Backbone과 Knockout의 경우 과다한 보일러플레이트, 대규모 어플리케이션에서의 한계를 가지고있고, AngluarJS의 경우 양방향 바인딩으로인해 규모가 커질수록 데이터 ↔ UI가 서로 영향을 주며 흐름이 복잡해지고, 변경 감지 비용이 커져 성능 / 디버깅 측면에서 한계가 드러났습니다.

이러한 배경에서 리액트는 단방향 데이터 바인딩을 제시하고, UI는 상태의 결과로만 결정되고, UI의 변경은 이벤트를 통한 상태를 바꾸는 방식으로 이루어지도록 설계되었습니다.

즉, 개발자가 어떻게 DOM을 바꿀지를 지시하느 대신, 무엇을 보여야하는지를 선언하면 실제 DOM 업데이트는 리액트가 알아서 처리하는 것이죠
