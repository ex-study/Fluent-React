### 초기 자바스크립트 개발 방식의 문제점

- 초반에는 정적 페이지가 많았고, 업데이트는 새 페이지를 썼다
- 웹의 성능이 좋아질수록 페이지가 즉시 업데이트되길 원하는 요구가 발생함
- ==유지 보수가 용이하고 / 재사용 가능하며 / 예측 가능한 방식으로 대규모 코드를 운용할 필요성이 생김==
- 문제점
  - 기존방식: 엘리먼트 탐색 API -> 이벤트 리스너 추가 -> 이벤트에 반응해 상태 업데이트 -> 이벤트 리스너 제거 후 상태 정리
  - 상태의 경우 엘리먼트에 직접 data-\* 속성을 추가하여 감지함
  - 목록에 새 항목을 추가하는 코드 -> 목록의 각 항목을 순차적으로 화면에 렌더링
- 왜 문제인가
  - 다른 클라이언트의 js에서 쉽게 접근할 수 있어 오류가 쉽게 생긴다
  - 다뤄야 할 요소가 js, html 양측에 존재하여 예측이 불가함
  - 목록의 각 항목을 순차적으로 DOM에 렌더링하면 계산 비용 면에서 비효율적

### 위를 해결하기 위한 JS 솔루션들이 나온 과정

#### jQuery

- 방식: 사용자 인터페이스에 데이터 바인딩 > 인터페이스 직접 업데이트
- 단방향 데이터 흐름
  - 양방향이 발생할수도 있는 구조
- 문제점
  - 코드의 어디서든 페이지 구조를 수정할 수 있음
  - 페이지 일부를 변경하면 다른곳에도 영향이 있어 문제 추적과 원인 파악이 어려워짐
  - 브라우저에 의존성이 커서 테스트가 어려움

#### Backbone - MVC 패턴

- 방식: 비즈니스 로직, 사용자 인터페이스, 사용자 입력을 분리
  - 모델: 데이터의 출처. 데이터 검색, 저장, 처리 담당.
  - 컨트롤러: 뷰에서 사용자 입력을 받고, 결과를 뷰에 반환.
  - 뷰: 사용자 인터페이스. 사용자 입력을 컨트롤러롤 전송.
- 인스턴스 생성 -> 인스턴스의 render 메서드를 호출해서 로직 테스트: 테스트가 더 쉬워짐
- 문제점
  - 대화형 요소가 많은 복잡한 사용자 인터페이스 관리 시 어려움. 구성 요소 간의 경계가 명확하지 않음
  - 양방향 데이터 바인딩: 뷰 <-> 모델 동기화 문제
  - 많은 보일러플레이트, 이벤트 관리 방식의 어려움, 중첩 구조의 어려움 등등

#### Knockout - MVVM

- 방식
  - 상태가 변경될 때마다 옵저버블(데이터의 출처 == 모델)과 바인딩(데이터를 소비/렌더링하는 UI == 뷰)으로 의존성을 추적함
  - 생성한 뷰 모델에 HTML 요소를 바인딩하여 상호작용함
- MVVM
  - 모델
    - 뷰, 뷰 모델 인식 불가
  - 뷰
    - 애플리케이션 로직X
    - 데이터 바인딩 메커니즘을 통해 자동으로 변경사항을 반영하고, 뷰 모델에 바인딩함
  - 뷰 모델
    - 모델 - 뷰 사이
    - 뷰에 바인딩할 데이터와 명령 제공
    - 모델의 데이터를 뷰에서 표시할 수 있도록 변환
    - 어떤 뷰가 자신을 사용하는지 알지 못함
  - MVC와의 차이
    - 결합과 바인딩
    - 컨트롤러가 없어서 -> 데이터 소유권이 더 명확함
    - 단반향 데이터 흐름 (뷰 -> 뷰모델 -> 모델)
- 장점
  - 데이터 바인딩을 통해 보일러플레이트 양 줄임
  - 테스트가 쉬움
- 단점: 여전한 보일러플레이트 코드, 뷰모델이 너무 커짐

#### 앵귤러JS

- 2010 구글 개발
- 양방향 데이터 바인딩: 모델 <-> 뷰 변경 사항을 자동으로 반영
- 모듈식 아키텍처 도입
  - 의존성들을 루트 app에 의존성 주입 패턴을 통해 주입
- 의존성 주입(DI)
  - 객체가 의존성을 전달받는 디자인 패턴
  - ex) 서비스를 의존성 주입을 통해 컨트롤러에 주입
- 장점
  - 구조화된 개발 -> 개발자의 작업 속도 향상
  - 양방향 데이터 바인딩을 통해 DOM 변경 처리
  - 의존성 주입, 모듈식 아키텍쳐 등
- 단점
  - 데이터 바인딩이 복잡한 대규모 애플리테이션에서 성능 문제
  - 새로운 개념 도입으로 인한 러닝커브
  - 커뮤니티 자체의 문제,,,,
  - 복잡한 문법
  - 타입 안정성 없고 디버깅 힘듬
  - 스코프 문제

#### 리액트

- 컴포넌트 기반 아키텍쳐
- 단방향 데이터 흐름
- 가상 DOM을 통한 직접적인 DOM 조작 최소화

### 리액트의 핵심 가치

#### 선언적 추상화

- 어떤걸 표현하고 싶은지 코드로 기술하면 -> 리액트가 그걸 이뤄줌 짱인데
- 자바스크립트만으로 표현할 수 있음
- 사용자 코드에서는 DOM의 식별자를 읽을 필요 없이, 고유한 리액트 엘리먼트를 사용하면 됨

#### 가상DOM

- 실제 DOM을 조작하지 않고, 이를 통해 UI를 업데이트함
- 가상DOM의 변경사항은 재조정이라는 과정으로 실제 DOM에 적용됨
  - 재조정: 새로운 가상DOM과 이전 가상DOM을 비교하는 과정
  - ex) 상태 변경, prop 변경 등 확인

#### 컴포넌트 모델

- DRY 개발
- 특정 컴포넌트만 성능 관리 작업(메모화 등) 및 추적(key props) 가능 -> 키잉
- 컴포넌트 조합 -> 합성

#### 불변 상태

- 애플리케이션 상태 -> 불변하는 값의 집합
- 각 상태 업데이트는 새롭게 독립된 스냅샷과 메모리 참조로 이루어짐
- 예측 가능한 UI 개발 가능

### 플럭스 아키텍쳐

- 클라이언트 측 웹앱 구축을 위한 아키텍처 디자인패턴
- 리액트 애플리케이션의 데이터 흐름 관리 패턴
- 액션
  - 새 데이터와 액션의 종류를 식별하는 객체
  - 디스패처를 통해 여러 스토어로 보내짐
- 디스패처
  - 액션을 받아서 애플리케이션에 등록된 스토어로 보냄
  - == 등록된 모든 콜백으로 해당 액션을 전송함
- 스토어
  - 객체의 상태를 관리하고, 액션을 처리하는 콜백 제공
  - 상태가 업데이트되면 -> 변경 이벤트 발생시켜 -> 뷰에 변경된 사항 알림
- 뷰
  - 리액트 컴포넌트
  - 스토어에서 변경 이벤트를 받고 의존성에 따라 업데이트
  - 시스템 상태를 업데이트하는 새로운 액션 생성
- 장점
  - 단일 정보 출처가 스토어에 저장: 데이터의 출처가 하나고, 그걸로만 모든 업데이트가 이루어짐
  - 관심사를 분리하면 독립적인 단위 테스트 가능
  - 관심사 분리가 명확
