JSX란 무엇인가 == 자바스크립트 XML의 약자

### 2.1 자바스크립트 XML

#### AJAX == 비동기 자바스크립트 & XML

- 현재 페이지를 새로고침하지 않고 서버 데이터를 가져오는 방식
  - 그때는 상태 변경 -> 새 페이지를 서버에서 받아 렌더링하는게 일반적
- UI 렌더링이 서버 -> 브라우저로 넘어오면서 JS로 DOM을 조작하는 코드 증가 -> 복잡성이 커짐

#### 비동기 업데이트 과정

- 브라우저에서 XMLHttpRequest 사용 -> HTTP를 통해 비동기 요청 (non-blocking) 보낼 수 있음
  - XMLHttpRequest: 브라우저에서 서버로 HTTP 요청을 보내는 오래된 JS API
- 요청에 대한 응답: 그 당시에는 XML, 지금은 JSON

#### JSX 간략 정의

- JSX는 개발자가 JS 내에서 HTML과 유사한 코드를 작성 가능하게 하는 JS용 구문 확장자
  - 메타에서 리액트와 사용하기 위해 개발 -> 다른 라이브러리/프레임워크에 채택
  - "확장 구문" == 컴파일을 통해 JS 코드로 변환됨

#### JSX와 HTML과의 차이

- 중괄호로 HTML 내에 JS 표현식을 넣음
- 속성 이름이 카멜케이스로 작성됨 ex) onclick(HTML), onClick(JSX)
- 엘리먼트 이름이 대문자로 작성 ex) div(HTML), Div(JSX)

### 2.2 JSX 장점

- 왜 쉽다고 느끼는가? HTML과 비슷하게 생김 == DOM의 구조와 유사함
- 데이터 소독: 새 엘리먼트를 생성할 수도 있는 문자들은 JSX를 컴파일할 때 딴걸로 바꿔줌.
  - ex) '<' -> &lt;
- 타입 안정성: 타입스크립트, JSDoc, propTypes를 통한 타입 안정성 향상
  - JSDoc: checkJs 옵션을 통해 타입검사 일부 가능
  - propTypes: 런타임 검사 라이브러리

```js
/**
 * @param {number} a
 * @param {number} b
 * @returns {number}
 */
function add(a, b) {
  return a + b;
}

add(1, 2); // OK
add("1", 2); // 타입 오류
```

- 컴포넌트 기반 아키텍처
- 광범위한 사용

<br>

### 2.3 JSX 약점

- 러닝커브
- JSX 코드를 실행하기 위해 JS 코드로 컴파일하는 개발 도구 필요
- JS + HTML -> 관심사 혼합
- JS 호환성 부족: 인라인 표현식 지원, 인라인 블록 미지원
  - `<div>{isLogin ? "로그인됨" : "로그인 필요"}</div>` -> 가능
  - `<div> {if (isLogin) { return <User /> }} </div>` -> 불가능

#### JSX 목적

- JS의 강력한 기능 유지 -> 반복, 계산, 인라인 실행
- 리액트 컴포넌트의 코드를 간단하게 작성 / 가독성 UP / 유지 보수 GOOD

### 2.4 내부 동작

#### 컴파일러

- 특정 규칙에 따라 소스 코드를 구문 트리로 변환하는 소프트웨어
  - 구문 트리: 코드의 구조를 나타내는 트리 자료 구조 ex) JS 객체
- 코드 컴파일 과정: 어휘 분석 -> 구문 분석 -> 의미 분석 -> 최적화 -> 기계어 생성

#### 컴파일러 과정

1. 토큰화: 문자열을 의미 있는 토큰으로 분해
   - 상태를 갖고 있는 토크나이저 (== 렉서)는 렉서 규칙으로 정규 표현식을 사용해 소스 코드에서 주요 토큰을 감지함 ex) 변수 이름, 객체 키 값 등
   - 감지한 토큰을 열거 가능한 값으로 변경 ex) const -> 0, let -> 1, ...
2. 구문 분석: 토큰을 구문 트리로 변환
   - 구문 분석기를 거치면 문자열은 JSON 객체가 됨
3. 코드 생성: 컴파일러가 추상 구문 트리 (AST)에서 기계어를 생성
   - 컴퓨터의 프로세서가 직접 실행할 수 있는 명령어로 AST의 코드를 변환
   - 기계어는 JS 엔진에 의해 실행

#### 컴파일러 종류

- 네이티브 컴파일러, 크로스 컴파일러, JIT 컴파일러, 인터프리터, ...
- 웹 브라우저를 비롯한 최신 환경에서는 JS 코드 실행에 JIT 컴파일러 사용
- JS 소스 코드가 중간 표현으로 변환 (바이트코드) -> 프로그램 실행 시 기계어로 동적 컴파일
  - 실시간 정보를 기반으로 최적화 가능 ex) 변수 종류, 자주 실행되는 코드 경로 등
  - 일부 엔진은 초기 실행 속도를 위해 빠른 컴파일 -> 자주 실행되는 코드만 더 최적화해서 컴파일 (hot code/path)

#### 런타임 (실행 환경)

- 프로그램이 실행될 때 필요한 환경 + API + 엔진의 집합
  - JS 런타임은 주로 웹 브라우저 ex) 크롬 웹 브라우저는 크로미움 런타임 제공
  - 서버측에서는 V8 엔진을 사용하는 Node.js 런타임 사용
- 브라우저 런타임은 JS 엔진에 window, document 객체 같은 컨텍스트 제공
  - 런타임마다 다른 컨텍스트 제공

### 2.4.2 JSX로 자바스크립트 구문 확장하기

- 구문 확장 == 새 구문(JSX)이 JS엔진에 도달하기 전에 처리하는 방식
  - 따라서 확장 언어로 작성된 코드를 이해할 수 있는 렉서 + 구문 분석기 필요 (1단계 + 2단계)
- JSX 전처리기(트랜스파일러)
  - 구문 트리를 사용해 기존의 JS 엔진이 이해할 수 있는 바닐라 JS 생성
  - Babel, TS, SWC, ..
- JSX 처리 과정
  1.  JSX 코드 작성
  2.  Babel / TypeScript / SWC 같은 트랜스파일러가 **렉싱 + 파싱 → AST 생성**
  3.  AST를 기반으로 **바닐라 JS 코드 생성**
  4.  변환된 JS가 번들에 포함되어 브라우저로 전달
  5.  브라우저 JS 엔진이 JS를 파싱 → 바이트코드 → 기계어 실행

<br>

### 2.5 JSX 프라그마

- JSX 프라그마: '<' 로 시작하는 지시어
  - 프라그마: 컴파일러에 특정 작업을 수행하도록 하는 지시어
- JS에서 위 프라그마는 인식 불가 (비교연산 제외)
- JSX에서 위 프라그마는 함수 호출로 변환
- JS의 지시어 예시
  - ex) use strict , use Client
- 솔직히 뭔말인지 모르겠음. 쨌든 함수 호출로 바뀐다는거 아니냐?
