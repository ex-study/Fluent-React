### 3.1 가상 DOM 소개

#### 가상DOM이란

- HTML 문서를 JS 객체로 만든거
  - DOM == 문서 모델 객체 == 브라우저 런타임의 문서 모델
- 가상 DOM == DOM의 가벼운 복사본
  - 실제 DOM은 노드 객체
  - 가장 DOM은 설명 역할을 하는 그냥 JS 객체

#### 리액트에서 가상DOM 사용

- 재조정: setState or 다른 매커니즘으로 UI 변경 지시 -> 가상 DOM 업데이트 -> 가상 DOM 변경 사항에 맞춰 실제 DOM 업데이트
- 실제 DOM의 업데이트가 느리고, 비용이 많이 들어 가상 DOM을 먼저 업데이트함
  - 레이아웃을 계산하는 리플로로 인해 DOM 상호 작용이 비효율적으로 동작
    - ex) 엘리먼트의 offsetWidth 읽기 -> 브라우저가 문서 전체/레이아웃 다시 계산
  - 반면 가상 DOM은 실제 페이지 레이아웃 변경X(리플로우 발생X)
  - 평범한 JS 객체이므로, 타 환경과 관계 X
- 가상 DOM 업데이트 후 비교 알고리즘으로 이전 버전과 새 버전의 차이점 식별
- DOM 업데이트에 필요한 최소한의 변경사항 선정 후 적용

<br>

### 3.2 실제 DOM

- 브라우저는 HTML 페이지를 읽고 -> 구문 분석해서 -> DOM(객체 모델)으로 변환
  - DOM은 노드와 객체의 트리 구조를 갖춘 JS 객체
  - 사용자가 페이지와 상호작용할 때마다 계속 업데이트
- 트리의 각 노드는 HTML 엘리먼트이고, JS로 조작할 수 있는 속성과 메서드 포함
- document.querySelector, document.getElementById: CSS 선택자를 기반으로 DOM에서 엘리먼트를 검색한다
  - 특정성: getElementById > querySelector

<br>

### 3.2.1 실제 DOM의 문제점

#### 성능

- DOM을 변경할 때마다 브라우저는 레이아웃을 다시 계산하고 그림 -> 많은 비용 발생
- 리플로우 발생 횟수 최소화 방법
  - getBoundingClientRect()로 레이아웃 읽기 및 쓰기를 일괄 처리 => 레이아웃 스레싱 최소화
    - getBoundingClientRect(): 한번 호출에 여러 레이아웃 속성을 검색
    - 레이아웃 스래싱: 레이아웃 속성 읽기/쓰기 시 발생하는 불필요한 재계산
    - 하지만 대기 중인 레이아웃 변경 사항이 있다면 정확한 값을 위해 리플로우가 발생할 수 있음
  - 속성의 값을 여러 번 읽어야 하는 경우 requestAnimationFrame API로 다음 애니메이션 프레임까지 속성 읽기를 미룰 수 있음
- CPU 사용량과 처리 시간 증가는 저전력 기기 사용자(스마트폰)에게 더 영향이 큼
- 이걸 react에서는 가상DOM을 둬서 해결

#### 브라우저 간 호환성

- 특정 DOM 엘리먼트와 속성을 지원하지 않는 브라우저가 있을 수 있다
- 이를 리액트의 합성 시스템으로 해결

##### SytheticEvent

- 브라우저의 기본 이벤트를 둘러싼 래퍼 객체 -> 여러 브라우저에서 일관성을 보장하기 위함
- 통합 인터페이스
  - 이벤트 속성에 접근하는 방법이 브라우저마다 다름
  - 그 차이를 추상화해 이벤트와 상호작용하는 방법 제시
- 이벤트 위임
  - 이벤트 리스너를 엘리먼트에 직접 추가하지 X, 루트에서 추가
  - 특정 엘리먼트에서 일부 이벤트 사용 불가능한 문제 방지
- event.nativeEvent를 이용해 네이티브 이벤트 접근 -> 추상화 + 유연함
- 그 밖에도 여러 이벤트에서 일관된 경험을 제공하도록 보장

<br>

### 3.2.2 Document Fragment

- DOM 노드를 저장하는 가벼운 컨테이너. 기본 DOM에 영향X
  - 진짜 DOM 노드. 메모리에만 있고, 렌더링 안됨. 다른 DOM 노드를 운반하는 컨테이너 역할.
- 문서 조각에 노드 추가 -> 여러 작업 진행 -> DOM에 추가
  - 실제 DOM에서는 리플로우, 리페인트 1번만 이루어짐 == 일괄 업데이트
- 메모리 효율성: 문서 조각에 추가된 노드는 실제 DOM에서 제거됨.
- 가상DOM과의 유사성
  - 일괄 업데이트, 효율적인 비교 알고리즘, 단일 렌더링

### 3.3 가상 DOM 작동 방식

<br>

### 3.3.1 리액트 엘리먼트

- React.createElement 함수를 통해 리액트 엘리먼트 생성
- `$$typeof` : 유효한 리액트 엘리먼트를 확인하는 Symbol. 종류 식별 표시자.
- type: 엘리먼트가 나타내는 컴포넌트의 종류 ex) div
- ref: 기본 DOM 노드에 대한 참조
- props: 컴포넌트에 전달된 속성과 props
- `_owner`: 해당 엘리먼트를 생성한 컴포넌트 추적
- `_store`: 엘리먼트에 대한 추가 데이터 저장

<br>

### 3.3.2 가상 DOM과 실제 DOM 비교

- React.createElement: 리액트에서 제공, 메모리에 새로운 가상 엘리먼트 생성
- document.createElement: dom API에서 제공, 메모리에 새로운 엘리먼트 생성 (DOM에 추가되기 전)
- createElement()의 인수
  - 첫번째 인수: 태그 이름 / 컴포넌트 이름
  - 두번째 인수: prop 객체 (리액트)
  - 추가 인수: children 엘리먼트 (리액트)
- 재조정 프로세스
  - 리액트 컴포넌트 렌더링 후 새 가상 DOM 트리 생성
  - 이전 가상 DOM 트리와 비교 후, 이전 트리를 업데이트하는데 필요한 최소 변경 횟수 계산

<br>

### 3.3.3 효율적인 업데이트

- 리액트 컴포넌트의 상태, prop 변경 시 새로운 리액트 엘리먼트 트리 생성
  - 이전 트리와의 비교는 재귀적으로 이루어짐
  - 이런 작업 == 디핑
- 리액트의 렌더링은 UI일관성을 위해 컴포넌트 + 모든 자손 컴포넌트를 리렌더링하는 방식
  - 불필요한 리렌더링이 되지 않게 최적화 기능 사용해서 잘 관리해라 ex) useMemo
