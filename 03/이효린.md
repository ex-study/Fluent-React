p.88
> 리플로가 발생하면 성능에 영향을 주고 직접적인 DOM 상호 작용이 비효율적으로 동작할 수 있습니다.
- Reflow는 전체적인 프로세스가 어떻길래 비용일 이렇게 크고 성능이 좋지 않는걸까 ?
	브라우저과 화면을 그리는 과정은 다음과 같다.
	```
	HTML 파싱
	   ↓
	DOM Tree 생성
	   ↓
	CSS 파싱
	   ↓
	CSSOM 생성
	   ↓
	Render Tree 생성
	   ↓
	Layout (Reflow)
	   ↓
	Paint
	   ↓
	Composite
	```
	 여기서 Reflow는 Layout 단계에서 발생하는 작업이다. 각 DOM 요소의 크가, 위치, 관계를 개산한다.
	 예를 들어 
	```
	div width: 100px
	
	div margin
	
	div position
	
	font-size
	```
	위와 같은 값들이 변경되면 전체 레이아웃을 다시 계산해야한다.
	Reflow는 트리 전체에 영향을 줄 수 있는데, 
	```
	한 요소 변경
	→ 부모 영향
	→ 형제 영향
	→ 자식 영향
	```
	위와 같은 방식으로 연쇄적으로 Layout 계산이 발생하기 때문이다.

p.93
> 로직을 컴포넌트화 할 수 있다는 점, 그리고 ... (중략) ... 점에서 리액트의 가상 DOM이 제공하는 가치는 의심할 여지가 없습니다.
- 맥락상 보면 로직을 컴포넌트화 할 수 있다는 것이 리액트의 가상 DOM이 제공하는 가치라는데, 그렇다면 리액트가 아닌 곳에서는 로직을 컴포넌트화 할 수 없다는건가 ?
	아니다. 컴포넌트화 자체는 React의 전유물이 아니다.
	```
	function createButton(label) {
	  const button = document.createElement("button");
	  button.innerText = label;
	  return button;
	}
	```
	위 코드는 Vanilla JS 코드로 이 또한 컴포넌트다.
	
	그렇다면 React에서 이야기하는 차별점은 무엇인가 ? **UI 상태 관리 + 업데이트 방식**이 다른데,
	- React 이전: DOM을 직접 조작함
	- React: state -> UI
		- 즉 `UI = f(state)` 인 구조이다
		- 상태를 기반으로 UI를 관리하고, DOM 업데이트를 최적화 하는 것이 React의 가치이다.

p.95
> 또 읽기와 쓰기 작업을 종류별로 나눠 일괄 처리하면 레이아웃 스레싱 현상이 최소화됩니다.
- 레이아웃 스레싱은 무엇이며 내가 실제 브라우저에서 테스트 할 수 있는 부분인걸까 ?
	레이아웃 스레싱은 DOM 읽기와 쓰기가 반복되면서 Reflow가 계속 발생하는 현상이다.
```
for (let i = 0; i < elements.length; i++) {
  elements[i].style.width = box.offsetWidth + "px";
}
```
여기서 문제는

```
읽기 → offsetWidth // 브라우저에서 layout 계산
쓰기 → style 변경 // 브라우저에서 layout 계산 
읽기 → offsetWidth // 브라우저에서 layout 계산
쓰기 → style 변경 // 브라우저에서 layout 계산
```

해결 방법은 읽기 / 쓰기를 분리하는 것이다.
```
const width = box.offsetWidth; // 읽기 1번

for (let i = 0; i < elements.length; i++) {
  elements[i].style.width = box.offsetWidth + "px"; // 쓰기 여러번
}
```

> 브라우저가 레이아웃 계산을 이미 수행한 경우라면 requestAnimationFrame API를 사용해 다음 애니메이션 프레임까지 속성 읽기를 미룰 수 있습니다.
- requestAnimationFrame의 동작은 어떨까 ?
	```
	requestAnimationFrame(callback)
	```
브라우저의 **다음 repaint 직전에 callback 실행**된다.
```
	JS 실행
	↓
	Style 계산
	↓
	Layout
	↓
	Paint
	↓
	requestAnimationFrame callback
	↓
	다음 frame
```
주로 애니메이션이나 DOM 업데이트 batching이다.
브라우저 프레임 타이밍에 맞춰서 실행할 수 있고, setTImeout보다 애니메이션을 부드럽게 사용할 수 있다.

p.102
> 문서 조각은 DOM 노드를 저장하는 가벼운 컨테이너입니다.
- '문서 조각'이라는 개념을 처음 들어서 공부해보고싶다
	**DocumentFragment란?**
	DOM 노드를 담을 수 있는 **가벼운 컨테이너**로
	- DOM Tree에 속하지 않음
	- reflow 발생 안함
	예시)
	```
	const fragment = document.createDocumentFragment();

	for (let i = 0; i < 1000; i++) {
	  const li = document.createElement("li");
	  li.textContent = i;
	  fragment.appendChild(li);
	}
	
	ul.appendChild(fragment);
	```
	DocumentFrame의 경우 DOM 업데이트가 1번만 발생한다.

p.103
> 문서 조각에 추가된 노드는 문서의 실제 DOM에서 제거됩니다.
- 굳이 실제 DOM에 있는 노드를 제거할 필요가 있을까 ?
	 **DocumentFragment가 특별해서 제거하는 것이 아니라, DOM API의 기본 동작이 “노드를 이동(move)”이기 때문**, 즉 -> 제거하는 것이 아님
	DOM 노드는 **동시에 두 개의 부모를 가질 수 없다.** 그래서 어떤 노드를 appendChild()로 다른 곳에 추가하면 브라우저는 내부적으로 다음과 같이 처리한다.
	- 기존 부모에서 제거 → 새로운 부모에 추가
	- fragment.appendChild(node)
		위 코드를 실행하면 브라우저는 
		```
	node.parentNode.removeChild(node)
	fragment.appendChild(node)
		```
위와 같은 방식으로 노드를 복사하는 것이 아니라, 이동시키게 한다.
따라서 **“굳이 제거한다”기보다는 DOM 트리 구조의 규칙 때문에 자연스럽게 이동되는 것**이라고 이해하는 것이 맞다.

> 문서 조각은 활성화된 문서 DOM트리에 속하지 않으므로 문서 조각을 변경해도 실제 문서에는 영향을 주지 않으며 ... (중략)
- 활성화된 문서 DOM트리는 실제 브라우저 DOM을 의미하는 것인가 ? 그렇다면 실제 DOM에서 제거된다는 윗 문장의 의미는 뭐지 ?
	여기서 말하는 **활성화된 문서 DOM 트리(active document DOM tree)** 는 브라우저가 실제로 렌더링에 사용하는 **document에 연결된 DOM 트리**를 의미한다.
		```
```
document
 └ body
     └ div
         └ ul
             └ li
```

이 트리는 브라우저가 **레이아웃 계산과 렌더링을 수행하는 실제 DOM**이다. 반면 **DocumentFragment**는 이 트리에 연결되지 않은 **독립적인 컨테이너**이다. 즉 document에 포함되지 않은 **off-DOM 상태의 노드 집합**이라고 볼 수 있다. 그래서 문서 조각 내부에서 노드를 추가하거나 수정해도 그 노드들이 **아직 document 트리에 연결되어 있지 않기 때문에 실제 화면에는 영향을 주지 않는다.**
한편 책에서 말하는 `문서 조각에 추가된 노드는 문서의 실제 DOM에서 제거됩니다.` 라는 문장은 **노드의 이동(move) 동작**을 설명하는 것이다. 
DOM 노드는 동시에 두 개의 부모를 가질 수 없기 때문에 `fragment.appendChild(node)`
를 실행하면 브라우저는 내부적으로 기존 DOM 트리에서 제거 → DocumentFragment로 이동을 수행한다.
